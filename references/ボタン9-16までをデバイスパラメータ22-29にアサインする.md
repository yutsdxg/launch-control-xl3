## トライ&エラーの履歴1
1. 初期実装
    - 目的: ボタン9-16でデバイスパラメータ22-29をトグル
    - 実施:
        - DeviceComponent に set_parameter_toggle_buttons を追加
        - [mappings.py](app://-/index.html# "mappings.py") で parameter_toggle_buttons: daw_mixer_buttons を接続
        - DEVICE_BANK_SIZE を 21 -> 29 に拡張
    - 結果: エラーなしだが無反応
2. パラメータ取得ロジックの改善
    - 仮説: 22-29番の参照に到達できていない
    - 実施:
        - _parameter_provider.parameters / _bank.parameters が不足時に
        - device.parameters（先頭 Device On 除外）からバンク計算するフォールバック追加
    - 結果: 依然無反応
3. ボタン競合回避（専用入力化）
    - 仮説: daw_mixer_buttons が既存機能（mute/select）と競合
    - 実施:
        - [elements.py](app://-/index.html# "elements.py") に Daw_Mixer_Buttons_Raw（private）追加
        - [mappings.py](app://-/index.html# "mappings.py") を parameter_toggle_buttons: daw_mixer_buttons_raw に変更
    - 結果: 依然無反応
4. 受信方式変更（register_slotの形を変更）
    - 仮説: ボタン行列 value の受け方（引数シグネチャ）が不一致
    - 実施:
        - 個別ボタンslot方式 -> 行列全体slot方式に変更
        - [(value, x, y, ...)](app://-/index.html# "(value, x, y, ...)") 前提でindex算出
    - 結果: 依然無反応（この時点で「マッピングキー自体が無視されている」可能性が高まる）
5. 追加調査途中で中断

- 次の仮説:
    - DeviceComponentBase が未知キー parameter_toggle_buttons をそもそもバインドしていない
    - その場合 set_parameter_toggle_buttons は呼ばれない
- 状態: 調査中にユーザー中断

**今回の学び（再発防止ポイント）**

- [mappings.py](app://-/index.html# "mappings.py") にキーを足す前に「ベースコンポーネントが受け付けるキー名」を先に確認する。
- 反応しないときは早期に「イベントが来ているか」を可視化（notify/log/一時LED）して、入力側と処理側を分離して切り分ける。
- 既存コンポーネント拡張では、controls デスクリプタ方式（ButtonControl/ButtonMatrixControl）に寄せる方がマッピングとの整合が取りやすい。

## トライ&エラーの履歴2
1. 初回実装
    - 変更: [device.py](app://-/index.html# "/Users/yuts/Data/Dev/launch-control-xl3/Launch_Control_XL_3_Custom/device.py") にトグル処理を直接追加、[mappings.py](app://-/index.html# "/Users/yuts/Data/Dev/launch-control-xl3/Launch_Control_XL_3_Custom/mappings.py") で parameter_toggle_buttons を Device に追加。
    - 仮説: DeviceComponent に独自 setter を足せばそのまま動く。
    - 結果: 効かない。
    - 学び: Device 既存コンポーネントへの独自レイヤーキー追加は、想定どおりバインドされない可能性がある。
2. 専用コンポーネント化（第1版）
    - 変更: [device_toggle.py](app://-/index.html# "/Users/yuts/Data/Dev/launch-control-xl3/Launch_Control_XL_3_Custom/device_toggle.py") 新規作成、[__init__.py](app://-/index.html# "/Users/yuts/Data/Dev/launch-control-xl3/Launch_Control_XL_3_Custom/__init__.py") に登録、[mappings.py](app://-/index.html# "mappings.py") の Encoder_Modes > daw_control へ追加。
    - 仮説: 独立コンポーネントなら確実に受け取れる。
    - 結果: 効かない。
    - 学び: control_list(ButtonControl) と ButtonMatrixElement の組み合わせが不安定/不一致の可能性。
3. イベント受信方式変更（matrix value 直受け）
    - 変更: [device_toggle.py](app://-/index.html# "device_toggle.py") を set_toggle_buttons + register_slot("value") に変更。
    - 仮説: matrix の value(x,y) を直接受ければ安定する。
    - 結果: まだ効かない。
    - 学び: 受信方式だけでは解決せず、所有権/競合やマッピング文脈の問題が残る。
4. mute/select 競合回避（専用 private matrix）
    - 変更: [elements.py](app://-/index.html# "/Users/yuts/Data/Dev/launch-control-xl3/Launch_Control_XL_3_Custom/elements.py") に Device_Toggle_Buttons（CC45-52の複製matrix）追加、[mappings.py](app://-/index.html# "mappings.py") 側で Device_Toggle はそちらを参照。
    - 仮説: 同一物理CCでも別要素に分離すれば競合を避けられる。
    - 結果: まだ効かない。
    - 学び: matrix複製でも解消しないケースがある（モード制御/優先度/所有権の影響が強い）。
5. パラメータ番号ズレ修正トライ
    - 変更: TOGGLE_PARAMETER_START_INDEX を 22 -> 21 -> 22 と調整。
    - 仮説: 0/1始まり解釈のズレ。
    - 結果: 一時的に「ボタン9でパラメータ21が少し動く」症状。
    - 学び: オフセットだけでなく、そもそも正しい対象へ安定バインドできていない。
6. モード依存の切り離し
    - 変更: Device_Toggle を Encoder_Modes 配下からトップレベルマッピングへ移動。
    - 仮説: daw_control 有効時だけ、という条件が発火を阻害している可能性。
    - 結果: 改善せず。
    - 学び: モード条件単体は主因ではない。
7. 現在の実装（個別ボタン直結）

- 変更: [elements.py](app://-/index.html# "elements.py") で CC45-52 を Device_Toggle_1..8_Button として個別追加、[device_toggle.py](app://-/index.html# "device_toggle.py") は ButtonControl 8本で1対1処理、[mappings.py](app://-/index.html# "mappings.py") で1対1マップ。
- 仮説: matrix経路を捨て、最短経路（個別ボタン）なら誤配線/競合を避けられる。
- 状態: 反映済み、再読み込み後の実機確認待ち。
